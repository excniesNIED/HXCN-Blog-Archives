## 导语

最近一段时间，Hermes Agent 在 X 和 GitHub 上被反复提起。这个项目从 2026 年 2 月开源后增长很快，到了 4 月上旬，社区讨论已经不只是“又一个 Agent 框架”，而是很多 OpenClaw 用户开始认真考虑：要不要换。

原因并不玄。对不少技术爱好者来说，OpenClaw 最吸引人的地方一直是能干活，能接工具，能接消息通道，也能长期在线；但真正用久了，麻烦也会一点点冒出来，比如升级后要补兼容、任务做到一半突然停住、上下文越跑越大、成本跟着往上抬。Hermes 之所以被讨论，不是因为它会说什么漂亮概念，而是因为它更像一套能长期跑下去的个人 Agent 系统。

![插图：开发者对比 OpenClaw 与 Hermes 的升级体验](./assets/6E84E04BF785493F4A6BF3B209A591E4.jpg)

从公开讨论看，很多人转向 Hermes，关注的也都是很实在的问题：它会不会自己中断，记忆能不能跨会话保留下来，长任务到底稳不稳，token 花得值不值。还有人提到，Hermes 放在常开设备上跑一整天，和本地模型、OpenRouter、Telegram 这类通道配合起来，比以前省心得多。

当然，迁移并不等于 OpenClaw 一下子没用了。更准确地说，Hermes 让一批用户看到了另一种路线：不是继续把 Agent 做得更热闹，而是先把记忆、执行和维护成本这几件事理顺。

## 爱马仕 Hermes 是什么

如果把大模型看成“会说话、会推理的大脑”，那真正决定它怎么行动的，往往不是模型本身，而是外面那层调度、记忆、工具和权限控制。Hermes Agent 做的，主要就是这一层。

它的核心思路很直接：让 Agent 不是一次性用完的工具，而是一个会积累经验的长期系统。你给它任务，它不只是执行；任务做完之后，它还会把过程中可复用的方法整理成技能，把和你相关的偏好留在记忆里，下一次再碰到相近问题时，尽量少从头摸索。对普通用户来说，这意味着它不只是“更聪明”，而是“越用越顺手”。

Hermes 这套设计也很适合本地优先的使用方式。它可以跑在家里的 Mac mini、小型 VPS、Docker 容器，或者接本地模型做低成本推理。数据、记忆、技能尽量留在自己的设备和私有环境里，而不是散在某个平台账号后面。对于想拿它看新闻、做日程整理、查资料、跑一些轻自动化任务的人来说，这种可控感其实比参数表更重要。

## OpenClaw 和 Hermes 的区别

两者都属于“把模型变成能做事的 Agent”这一路，只是侧重点不一样。OpenClaw 更像一个能力铺得很开的执行平台，工具多、生态热闹、接入方式也丰富；Hermes 则更像一个收得更紧的个人助理，重点放在长期记忆、技能沉淀和持续运行的稳定性上。

很多人决定迁移，往往不是因为 OpenClaw 功能少，而是因为日常使用成本偏高。社区里常见的抱怨包括：更新后要额外修问题，任务做到一半莫名停下，上下文膨胀得快，插件和权限边界需要自己花精力盯着。甚至连“明明说了先确认再操作，结果它直接开删”这种事故，也成了大家反复拿来调侃的例子。

![插图：关于 OpenClaw 误操作的社区吐槽](./assets/aba5968c-ca5c-4e95-a320-7d125ebe36eb.png)

Hermes 更受欢迎的地方，则集中在另一面：遇到错误时更愿意继续尝试，直到任务明确成功或失败；上下文控制得更紧，长对话里的 token 压力没那么夸张；跨会话记忆更稳定，不需要反复把同一套背景再讲一遍。对小白和轻度折腾用户来说，这些改进听起来不花哨，却最接近日常体验。

![插图：用户总结 Hermes 在续做任务和 token 控制上的体验](./assets/C479815ADCD7AA6F4E7B94F24FE83678.jpg)

但也要说清楚，Hermes 不是“全方面碾压”。如果你特别依赖 OpenClaw 现成的插件生态、多平台集成，或者已经围绕它搭出一整套工作流，那么迁移前还是得评估一下替换成本。只是对于多数想把 Agent 当成个人助手长期挂着用的人来说，Hermes 的方向更像是在减负，而不是继续加功能。
